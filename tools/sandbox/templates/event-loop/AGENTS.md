# {{title}} — sandbox

> Учебная песочница. Ты — и тьютор, и инженер: объясняй как работает JS event loop — call stack, очередь микротасок (Promise.then / queueMicrotask), очередь макротасок (setTimeout) — и почему именно в таком порядке появляется вывод. Общие правила проекта: ../../AGENTS.md

## Что здесь изучается

- **Call Stack** — синхронный код выполняется до конца, прежде чем движок взглянет на очереди.
- **Microtask Queue** — `Promise.then()` и `queueMicrotask()` попадают сюда; движок сливает **все** микротаски прежде чем взять следующую макротаску.
- **Macrotask Queue** — `setTimeout`, `setInterval`; берётся **одна** задача за проход event loop.
- **Порядок вывода** — ключевое «озарение»: sync 1 → sync 2 → микротаски (все) → macrotask.

## Что уже в скаффолде

- Кнопка **Run** выполняет реальный сценарий через `setTimeout` / `Promise.resolve().then` / `queueMicrotask` и добавляет каждую строку в сигнал `outputLines` в том порядке, в котором JS их вызвал — никакой фальсификации.
- Каждая строка вывода помечена бейджем фазы: **sync** / **microtask** / **macrotask**.
- Диаграмма **модели event loop** — три колонки (Call Stack, Microtask Queue, Macrotask Queue) с легендой правила цикла.
- Построитель **дополнительных задач**: кнопки «+ setTimeout», «+ Promise.then», «+ queueMicrotask», «+ sync log» добавляют задачи к сценарию перед запуском.

## Ловушки / на что смотреть

- **Microtasks перед macrotasks**: частая ошибка — думать, что `setTimeout(fn, 0)` выполнится сразу. Нет: сначала сливаются **все** микротаски.
- **queueMicrotask vs Promise.then**: оба — микротаски, порядок между ними зависит от порядка регистрации.
- **Порядок нескольких setTimeout**: без разных задержек порядок гарантирован только внутри одной очереди, но между ними может быть один проход цикла.
- **Angular + zone.js**: в проекте может быть zone.js, который патчит `setTimeout` и запускает change detection после каждой макротаски — это не меняет порядок event loop, но добавляет лишние CD-проходы.
