# {{title}} — sandbox

> Учебная песочница. Ты — и тьютор, и инженер: объясняй как работает JS event loop — call stack, очередь микротасок (Promise.then / queueMicrotask), очередь макротасок (setTimeout) — и почему именно в таком порядке появляется вывод. Общие правила проекта: ../../AGENTS.md

## Что здесь изучается

- **Call Stack** — синхронный код выполняется до конца, прежде чем движок взглянет на очереди.
- **Microtask Queue** — `Promise.then()` и `queueMicrotask()` попадают сюда; движок сливает **все** микротаски прежде чем взять следующую макротаску — в том числе **между** макротасками.
- **Macrotask Queue** — `setTimeout`, `setInterval`; берётся **одна** задача за проход event loop.
- **Ключевое «озарение»** — сценарий содержит доказательство «микротаски сливаются между макротасками»: `timeout A` планирует `Promise.then` внутри себя, и эта микротаска выполняется **до** `timeout B`.

## Как устроена демка (архитектура)

- **Реальный прогон, записанный.** Run выполняет НАСТОЯЩИЙ сценарий (`setTimeout` / `Promise.resolve().then` / `queueMicrotask` — ничего не подделывается) и пишет каждый шаг в `RunRecorder` (`../../shared/learning`): kind (`sync`/`microtask`/`macrotask`/`info`), метка, однострочное «почему» и снимок `{ stack, micro, macro }`.
- **Зеркала очередей.** Рядом с настоящими вызовами планирования код ведёт зеркальные массивы `mirrorMicro`/`mirrorMacro`: push при регистрации колбэка, удаление — из самого колбэка, когда движок его реально запустил. Порядок выполнения задаёт движок; зеркало лишь фиксирует его состояние для снимков.
- **Степпер-реплей.** После завершения прогона (все макротаски отработали, очереди пусты) шаги копируются ОДИН раз в `replaySteps`, и `<sb-stepper>` проигрывает запись: Шаг / Авто / Сначала. Три колонки (Call Stack / Microtask Queue / Macrotask Queue) привязаны к снимку ТЕКУЩЕГО шага; позиция −1 показывает состояние до запуска.
- **Карточка-предсказание.** `<sb-experiment-card>` спрашивает «какая строка вывода появится третьей» до запуска; `actualIndex` вычисляется из РЕАЛЬНО записанных шагов после прогона (добавленные задачи могут изменить ответ — он пересчитывается, а не захардкожен).
- **Построитель дополнительных задач**: «+ setTimeout», «+ Promise.then», «+ queueMicrotask», «+ sync log» дописывают задачи в конец сценария (после `sync 2`). **Reset** очищает запись, задачи и карточку.

## Попробуй (эксперименты)

- [ ] В карточке «Предскажи результат» выбери вариант, нажми **Run** — проверь себя. Правильный ответ без дополнительных задач: `promise .then` (сначала весь синхронный код, потом ВСЕ микротаски, потом макротаски).
- [ ] Прощёлкай степпером весь реплей и поймай момент-доказательство: на шаге `timeout A` в Microtask Queue появляется «микротаска из timeout A», а `timeout B` всё ещё ждёт в Macrotask Queue — следующий шаг именно микротаска, не timeout B.
- [ ] Найди шаг `скрипт закончился`: стек пуст, обе очереди заполнены — это момент, когда цикл поворачивается к очередям.
- [ ] Добавь «+ sync log», предскажи заново и запусти — правильный ответ карточки сменится на «другая строка»: sync-задачи печатаются ещё в синхронном блоке, сразу после `sync 2`.
- [ ] Добавь несколько «+ Promise.then» и «+ queueMicrotask» — в реплее все они сливаются **до** `timeout A`.
- [ ] Добавь «+ setTimeout» — в реплее он встаёт в Macrotask Queue **за** `timeout B` и выполняется последним.

## Ловушки / на что смотреть

- **Microtasks перед macrotasks**: частая ошибка — думать, что `setTimeout(fn, 0)` выполнится сразу. Нет: сначала сливаются **все** микротаски.
- **Микротаски между макротасками**: очередь микротасок сливается заново после КАЖДОЙ макротаски — поэтому микротаска из `timeout A` обгоняет `timeout B`.
- **queueMicrotask vs Promise.then**: оба — микротаски одной очереди, порядок между ними — порядок регистрации.
- **Снимки колонок** показывают состояние **после** шага: у шага `timeout A` его колбэк ещё на стеке, а запланированная им микротаска уже в очереди.
- **Приложение zoneless** (без zone.js): change detection не привязан к патчам таймеров; запись шагов буферизуется и сбрасывается в signal вне CD-прохода — не пиши в signals синхронно во время CD.
